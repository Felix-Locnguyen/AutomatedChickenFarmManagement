CÂU HỎI SỐ 1: Configuration là gì, lí do ra đời, tại sao lại cần sử dụng

    - Config (viết tắt của Configuration) là chương trình thiết lập và cấu hình
    cài đặt cho hệ thống của dự án có thể hoạt động hiệu và dễ bảo trì 
    
    - Lí do ra đời:
        + Trước đây người ta muốn thay đổi chỉ số, code của hệ thống thì phải 
        thay đổi code trực tiếp trong chương trình, gây mất thời gian và nhiều 
        công sức như
            > Dễ xảy ra lỗi khi đổi code
            > Phải xây dụng lại hệ thống
            > Yêu cầu kiến thức và hiểu cấu trúc dự án chuyên sâu để thay đổi
        => chính vì điều đó người ta tổng hợp lại thành 1 folder quản lí chung 
        để có thể để có thể dễ dàng thay đổi code dễ dàng

CÂU HỎI SỐ 2: Tại sao phải làm routes, lí do nó ra đời và lợi ích là gì???

    - Routes: là cách để định nghĩa các đường dẫn URL cho các trang web hoặc API trong
    dự án

    - Lí do ra đời:
        Trước đây mỗi URL tương ứng với một file vật lý trên server (ví dụ /trangchu.html, /lienhe.html).
        Khi dự án lớn lên, hàng trăm file nằm rải rác → loạn. Routes ra đời để tách URL khỏi file vật lý, 
        URL chỉ là "tên gọi", còn logic xử lý nằm trong code.
    
    - Lợi ích:
        + Quản lí và định tuyến các yêu cầu trong dự án một cách dễ dàng và hiệu quả hơn 
        + Cải thiện tính hoạt động và khả năng mở rộng của dự án 
        + Giảm thiểu thời gian và công sức khi quản lí và cập nhật các trang web
    
    - Nguyên lí hoạt động 
        + sơ đồ tổng quan
            Người dùng → Server kiểm tra URL → Chạy hàm xử lý (function) → Trả về kết quả → Hiển thị trang


            + Mô tả:
            > khi người dùng truy cập vào URL thì hệ thống sẽ gửi yêu cầu đến sever để kiểm tra xem đường dẫn
            có tồn tại hay không trong routes, nếu tồn tại sẽ truy cập được trang web nếu không thì sẽ bị lỗi 404

CÂU HỎI SỐ 3: Blueprint, request, jsonify là gì và tại sao chúng được dùng nhiều trong backend?
    - Blueprint:
        + Là công cụ chia nhỏ ứng dụng thành các phần độc lập, mỗi phần quả lí một nhóm routes riêng 
        + Lý do ra đời: Khi dự án lớn lên nếu để tất cả routes trong một file app.py thì file đó sẽ fafi và 
        có hàng nghìn dòng => khó quản lí và phát triễn thêm
            ví dụ không Blueprint
                @app.route('/api/a')
                @app.route('/api/a/a1')
                @app.route('/api/a/a1/a1.1')
                @app.route('/api/b')
                ...
            ví dụ có Blueprint
            Tách ra từng file 
                livestock_bp  → livestock/routes.py
                feed_bp       → feed/routes.py
                sensor_bp     → sensor/routes.py
                user_bp       → user/routes.py
        + tác dụng chính là chia nhỏ dự án thành nhiều module (file) đảm bảo dễ đọc, dễ bão trì, dễ mở rộng

    - Request:
        + Là công cụ đọc dữ liệu của người dùng gửi cho sever
        + ví dụ cách sử dụng
        # ví vụ về đăng nhập
        @app.routes('/app/login', methouds=['POST])
        def login()
            data = request.get_json()    # đọc dữ liệu join
            username = data['username']
            password = data['password]

            # hoặc lấy giá trị 
            farm_id = request.args.get('farm_id')
    
    - jsonify 
        + Là công cụ đóng gói dữ liệu thành dạng json và trả về cho người dùng
        + lý do ra đời: Sever xử lí xong cần trả kết quả, hiện tại web và mobile đều xử dụng JSON
        để trao đổi dữ liệu vì nó nhanh 
        + ví dụ 
            @app.route('/api/livestock')
            def get_livestock():
                data = {
                    "total": 150,
                    "healthy": 142,
                    "sick": 8
                }
                return jsonify(data)  # ← đóng gói và trả về

            # Người dùng nhận được:
            # {
            #     "total": 150,
            #     "healthy": 142,
            #     "sick": 8
            # }
        + tác dụng: chuyển dữ liệu list, dict,.. thành JSON chuẩn để trả về cho fontend/mobile app để đọc và hiển thị


        sơ đồ khi kết hợp Blueprint, Request và jsonify
              
            [ NGƯỜI DÙNG GỬI YÊU CẦU ]
                    ↓
                Blueprint           → xác định đúng file/module xử lý
                    ↓
                request             → đọc dữ liệu người dùng gửi lên
                    ↓
            [ XỬ LÝ LOGIC ]         → truy vấn DB, tính toán...
                    ↓
                jsonify             → đóng gói kết quả thành JSON
                    ↓
            [ TRẢ VỀ CHO NGƯỜI DÙNG ]

CÂU HỎI SỐ 4: flask_jwt_extended , create_access_token, jwt_required, get_jwt_identity là gì, lí do ra đồi và nguyên lí hoạt động ?
    - flask_jwt_extended (thư viện mẹ):
        + Là thư viện mở rộng của Flask dùng để xác thực người dùng bằng JWT (JSON Web Token)
        + Giúp xử lý các chức năng như:
            > Tạo token khi đăng nhập
            > Bảo vệ API (yêu cầu đăng nhập)
            > Lấy thông tin người dùng từ toke
        + JWT (JSON WEB TOKEN) cgiống như một “thẻ thông hành” — sau khi đăng nhập thành công, server cấp cho người dùng một token.
        Mỗi lần gọi API, người dùng chỉ cần gửi token này thay vì phải đăng nhập lại.
        + Lí do ra đời
            > Ngày xưa khi người dùng đăng nhập, sever lưu thông tin vào Ram khi đạt tới số lượng người dùng nhất định (ví dụ 1000 người)
            Ram sẽ đầy và dẫn tới sever sẽ chậm sinh ra nhiều vấn đề về trải nghiệm người dùng
            > JWT ra đời: khi người dùng đăng nhập, sever sẽ cấp Token và lưu vào máy người dùng, sever sẽ không phải lưu gì cả điều này 
            giúp giải phóng lượng lớn dữ liệu vào RAM giúp sever mượt mà hơn 
            => Thay vì server phải lưu trạng thái đăng nhập của từng người dùng, JWT chuyển việc đó sang phía client bằng cách sử dụng token.
            Điều này giúp hệ thống nhẹ hơn, dễ scale hơn và phù hợp với API hiện đại.
    
    - create_access_token:
        + Hàm tạo ra token sau khi người dùng đăng nhập
        
        @app.route('/api/login', methods=['POST'])
        def login():
        data = request.get_json()
        username = data['username']
        password = data['password']

        # Kiểm tra đúng mật khẩu
        if username == "admin" and password == "1234":
            
            # Tạo token, đính kèm thông tin người dùng vào
            token = create_access_token(identity=username)
            #                           ↑
            #              {{ gắn username vào trong token }}

            return jsonify({"token": token})
        
        return jsonify({"error": "Sai mật khẩu"}), 401

        + Cấu trúc Token 
            | eyJhbGciOiJIUzI1NiJ9 |.eyJzdWIiOiJhZG1pbiJ9 | .abc123xyz |
            |      Phần header     |      Phần data       |    chữ kí  |
    
    - jwt_required:
        + Có thể hiểu là người bảo vệ kiểm tra vé (token) có hợp lệ hay không rồi mới
        cho phép truy cập vào route:
        + Nguyên lí hoạt động:

            # Route KHÔNG có jwt_required → ai cũng vào được
            @app.route('/api/public')
            def public():
                return "Trang công khai, ai cũng xem được"


            # Route CÓ jwt_required → phải có token mới vào được
            @app.route('/api/livestock', methods=['GET'])
            @jwt_required()                  # ← bảo vệ cửa đứng đây
            def get_livestock():
                return "Danh sách vật nuôi — chỉ người đăng nhập mới thấy"
    
    - Các thông tin đầu ra của từng trường hợp 
        Không có token -> 401 Unauthorized (chưa đăng nhập)
        Token sai -> 422 Unprocessable (token không hợp lệ)
        Token hết hạn -> 401 Unauthorized (Token đã hết hạn)
        Token đúng -> vào trang web

        KIẾN THỨC THÊM
            [1] Nhóm status code trong web
                1xx Thông tin (ít dùng)
                2xx Thành công
                3xx chuyển hướng 
                4xx Lỗi phía client
                5xx lỗi phía server

            [2] các mã thành công (2xx)
                200: ok (Request thành công, Server trả dữ liệu bình thường)
                201: Created (Tạo dữ liệu thành công)
                204: No Content (Thành công nhưng không trả dữ liệu)

            [3] các lỗi phía client (4xx)
                400: Bad Request (request sai do thiểu file,sai format JSON,...)
                401: Unauthorized (chưa xác thực do chưa login hoặc token hết hạn)
                403: Forrbidden (Đã xác thực nnhưngkhoong có quyền truy cập)
                404: Not Found (Lỗi thường gặp nhất, không tìm thấy tài nguyên(API,ID) )
                405: Method Not Allowed (sai method(Get/POST/PUT/DELETE))
                409: Confilict (xung đột dữ liệu)
                422: Unprocessable Entity (Dữ liệu đúng forrmat nhưng sai logic / không hợp lệ)
            
            [4] các lỗi phía sever (5xx)
                500: Internal Sever Error (Lỗi sever, code bug, crash,...)
                502: Bad Gateway (Sever trung gian nhận phản hồi Lỗi)
                503: Service Unavaliable (Sever quá tải hoặc đang bảo trì)

    
    - get_jwt_identity
        + Hàm đọc thông tin người dùng dùng được nhúng bên trong token - giúp sever biết ai đang 
        gửi yêu cầu
        + Nguyên lí hoạt động:

            from flask_jwt_extended import jwt_required, get_jwt_identity

            @app.route('/api/my-farm', methods=['GET'])
            @jwt_required()
            def get_my_farm():
                
                # Đọc thông tin từ token
                current_user = get_jwt_identity()
                # current_user = "admin" (lấy từ identity khi tạo token)

                # Dùng thông tin đó để truy vấn đúng dữ liệu
                farm_data = Farm.query.filter_by(owner=current_user).all()
                
                return jsonify(farm_data)
        
        Sơ đồ quy trình 

            BƯỚC 1 — Đăng nhập:
                    [ Người dùng gửi username + password ]
                                ↓
                    [ Server kiểm tra đúng/sai ]
                                ↓ đúng
                    [ create_access_token → tạo token ]
                                ↓
                    [ Trả token về cho người dùng lưu lại ]


            BƯỚC 2 — Truy cập trang bảo mật:
                    [ Người dùng gửi yêu cầu + kèm token ]
                                ↓
                    [ jwt_required → kiểm tra token hợp lệ không ]
                                ↓ hợp lệ
                    [ get_jwt_identity → đọc "đây là user nào" ]
                                ↓
                    [ Trả đúng dữ liệu của user đó ]

CÂU HỎI SỐ 5: werkeug.security, generate_password_hash, check_password_hash là gì ?, Lí do ra đời?, Nguyên lí hoạt động?
    - werbeug.security:
        + là một phần thư viện của werkzeug - bộ công cụ nền tẳng của Flask chuyên xử lí các vấn
        đề bảo mật, mật khẩu 
        + lí do ra đời:
            > Ngày xưa lưu mật khẩu thẳng vào Database dẫn tới việc nếu hacker truy cập được vào database thì việc lấy mật khẩu
            rất dễ 
            > werkeug.security ra đời, hệ thông sẽ mã hóa password khiến cho hacker không đọc được mật khẩu
    
    - generate_password_hash:
        + Hàm mã hóa mật khẩu 
            > Biến mội thứ thành một chuỗi ngẫu nhiên không thể đọc được
            > Nguyên lí hoạt động:

            from werkzeug.security import generate_password_hash

            mat_khau_that = "1234abcd"
            mat_khau_da_bam = generate_password_hash(mat_khau_that)

            print(mat_khau_da_bam)
            # $pbkdf2-sha256$29000$randomSalt$chuoiKyTuNgauNhien...
            # ↑ Không ai đọc được kể cả chính bạn!
    
    - check_password_hash:
        + Hàm kiểm tra mật khẩu (đối chiếu với mật khẩu người dùng nhập ) bằng việc giải mã ngược
        + Nguyên lí hoạt động 

            from werkzeug.security import check_password_hash

            ket_qua = check_password_hash(hash_trong_db, mat_khau_nhap)
            # Hàm này biết cách "giải mã ngược" để so sánh đúng cách

CÂU HỎI SÔ 6: flask_sqlalchemy là gì, lí do ra đời, nguyên lí hoạt động
    - flask_sqlalchemy: là một thư viện giúp bạn làm việc với database bằng code Python thay vị chạy trên SQL thuần
    - Lí do ra đời:
        > Lúc chưa có SQLALchemy:

            # Muốn lấy danh sách vật nuôi phải viết SQL thuần
            cursor.execute("SELECT * FROM livestock 
                            WHERE farm_id = 1 
                            AND status = 'healthy'
                            ORDER BY created_at DESC")
            
            # Muốn thêm vật nuôi mới
            cursor.execute("INSERT INTO livestock 
                            (name, species, weight, farm_id) 
                            VALUES (?, ?, ?, ?)", 
                            ("Bò 01", "Bò", 150, 1))
  
            # Muốn cập nhật
            cursor.execute("UPDATE livestock 
                            SET weight = 160 
                            WHERE id = 1")

            → Dài dòng, dễ sai cú pháp, dễ bị tấn công SQL Injection

        > Lúc có SQLALchemy:

            # Lấy danh sách vật nuôi
            Livestock.query.filter_by(farm_id=1, status='healthy')\
                           .order_by(Livestock.created_at.desc()).all()

            # Thêm vật nuôi mới  
            con_bo = Livestock(name="Bò 01", species="Bò", weight=150)
            db.session.add(con_bo)

            # Cập nhật
            con_bo.weight = 160
            db.session.commit()

            * lí do nên sử dụng
                _ KHó bị hack vì không bị lộ câu lệnh SQL 
                _ Có thể lưu biến và sử dụng liên tục thay vì code lại câu lệnh
                _ Không cần nhớ cú pháp SQL
                _ Đổi database không cần phải fix lại code
